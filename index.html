SLIDE 5
Query Processing Workflow
End-to-End Query Processing Flow
1
Query Input

User submits query: "What is machine learning?"

2
Hybrid Retrieval

Vector Search: Query embedding → Cosine similarity → Top-K = 10

Keyword Search: BM25 scoring → Term frequency → Top-K = 10

Fusion: Reciprocal Rank Fusion combines both results

3
Reranking

BGE Reranker Model (BAAI/bge-reranker-base)

Cross-encoder scoring between query and each chunk

Output: Top-3 most relevant chunks

4
Context Assembly

Combine top chunks into context for LLM

Context = """
[Source 1] Machine learning is a subset of AI...
[Source 2] ML algorithms learn from data patterns...
[Source 3] Common types: supervised, unsupervised, reinforcement...
"""
5
LLM Generation

Local Models: llama3, mistral, mixtral (via Ollama)

Cloud Models: gpt-3.5-turbo, gpt-4 (via OpenAI API)

6
Response
{
  "answer": "Machine learning is a subset of artificial intelligence..",
  "sources": ["doc1.pdf", "doc2.pdf"],
  "confidence": 0.92,
  "retrieval_time": 0.45,
  "generation_time": 1.23
}

Chunk 1: "Machine learning is a subset of AI. It enables systems..."
────────────────────────────────────────────────────────────────────

────────────────────────────────────────────────────────────────────

Embedding Phase (embedding.py)
for chunk in chunks:
    embedding = model.encode(chunk)  # 384-dim vector
    cache.set(hash(chunk), embedding)  # Cache for reuse

Output: embeddings array (n_chunks × embedding_dim)

  • Build FAISS index from embeddings
  • Save to RAG_Logic/data/DBVectorial/vector_index.pkl
  • Build BM25 index for keyword search

🔄
For Each Configuration
  1. Re-index documents with new chunking parameters
  2. For each evaluation question:
    • Retrieve documents using current retrieval config
    • Generate answer using LLM
    • Compare with expected answer
    • Calculate metrics: accuracy, semantic similarity, precision/recall
  3. Aggregate scores across all questions

SLIDE 11
Performance & Caching Architecture
Multi-Layer Optimization
1
Layer 1: Embedding Cache

Key: hash(text_chunk)

Value: embedding (np.ndarray)

TTL: 7 days (configurable)

Storage: File-based (pickle)

Hit Rate: Typically 60-80% after warmup

2
Layer 2: Reranking Cache

Key: hash(query + document_text)

Value: relevance_score (float)

TTL: 24 hours

Storage: Memory + File

Benefit: Avoids expensive cross-encoder inference

3
Layer 3: Vector Index Persistence

Startup: Load index from DBVectorial/vector_index.pkl

Runtime: In-memory FAISS index for fast search

On Change: Save updated index to disk

Benefit: 10-100x faster startup time

4
Layer 4: Model Lazy Loading

Models only loaded when first needed

Singleton pattern per model type

GPU memory optimization through shared instances

Benefit: Reduced memory footprint, faster startup

Getting Started & Commands
Quick Start Guide

📦 Installation

# Clone repository
git clone https://github.com/yourusername/AUTO_RAG.git
cd AUTO_RAG

# Install Python dependencies
pip install -r requirements.txt

# Install frontend dependencies
cd frontend && npm install && cd ..

# Install Ollama (optional, for local LLMs)
curl -fsSL https://ollama.com/install.sh | sh
ollama pull llama3

🚀 Running the System

# Start API server (Terminal 1)
python main_api.py

# Start frontend (Terminal 2)
cd frontend && npm start

# OR use CLI mode
python cli_optimized.py

⚡ Common Commands

# Run experiments
python run_experiments_comprehensive.py

# Manage models
python model_manager.py list
python model_manager.py install llama3
python model_manager.py test llama3

# Check system status
curl http://localhost:8000/docs

# Quick query test
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"query": "What is RAG?"}'

🔧 Environment Variables

export RAG_EMBEDDING_MODEL="sentence-transformers/all-MiniLM-L12-v2"
export RAG_OLLAMA_MODEL="llama3"
export RAG_CHUNK_SIZE=400
export RAG_OVERLAP=50
export RAG_TOP_K=5
export RAG_LOG_LEVEL=INFO
export RAG_FORCE_CPU=false
